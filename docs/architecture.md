Тут описание архитектуры проекта, обоснование принятых решений (API vs local inference), план развития до сервиса (API, idempotency, storage, async, мониторинг и т.д.).

Решение построено как двухэтапный пайплайн: ocr_service извлекает текст и confidence из изображения, затем extraction_service нормализует текст и по правилам/контекстным маркерам выделяет сущности Фамилия, Имя, Отчество. main.py оркестрирует шаги и возвращает структурированный JSON-результат с диагностической информацией (confidence, ошибки распознавания).

Подтвержденный рабочий алгоритм, зафиксированный в `notebooks/research.ipynb`, реализован как два OCR-прохода + простое извлечение ФИО по правилам. Сначала изображение загружается и при необходимости уменьшается, если его разрешение слишком большое, чтобы стабилизировать качество распознавания и ограничить вычислительные затраты. Затем выполняется первый OCR по всему изображению, в результатах которого по нечеткому совпадению находится якорный текст `Собственник`, после чего вокруг него строится crop области, где ожидается блок с данными владельца.

На втором этапе OCR запускается уже только на найденном crop, что уменьшает шум от остальных полей документа и повышает вероятность корректного чтения ФИО. Полученный текст далее обрабатывается набором простых правил: токены нормализуются, отбрасываются латинские артефакты OCR и служебные слова, после чего из строк после якоря выбираются три последовательных токена, интерпретируемые как `Фамилия`, `Имя` и `Отчество`. Такой подход хорошо подходит для MVP: он прост, воспроизводим и опирается на устойчивую структуру СТС.

Лучшая точность извлечения может быть достигнута за счет препрпоцессинга изображений (CLAHE, doc crop, etc...). Переход на VLM модель для извлечения сущностей из изображения (PaddleOCR-VL/DeepSeek-OCR). Разбиение изображения на чанки.

В качестве базового решения выбраны локальные модели, чтобы обеспечить полностью автономный и воспроизводимый запуск без API-ключей и внешних зависимостей. Архитектура построена через абстракции провайдеров OCR и extraction, поэтому при необходимости можно переключиться на API-провайдеров для повышения качества/скорости без изменения бизнес-логики.
